
# How to Scrum

このドキュメントは、私たちのプロジェクトをスクラムで進めるための手引書です。全員でこの進め方を共有し、実践しながら私たち自身のやり方を作っていきましょう。

## 登場人物

* **スクラムチーム**: プロダクトの価値を創造する自己管理型のチーム。
  * **スクラムマスター (SM)**: スクラムが正しく導入され、機能するように支援する。チームの障害物を取り除くことにも責任を持つ。
  * **プロダクトオーナー (PO)**: プロダクトの価値を最大化することに責任を持つ。プロダクトバックログ(PBL)の管理者。
  * **開発者 (Developers)**: モノ（利用可能なインクリメント）を作成する専門家。3〜9名程度で構成されるのが一般的。
* **ステークホルダー**: プロダクトにお金を出す人、ビジネス部門の上位者、ユーザーなど、プロジェクトの成果に利害関係を持つ人たち。

* [ ] **Whyの共有**: チーム全員で「なぜやるのか」を話し合い、**プロダクトゴール**（長期目標）を設定する。

* [ ] **相対見積もり**: プランニングポーカーなどで各PBIの「大きさ」を見積もる。

## プロジェクト開始までの手順

### 1. 「なぜ」やるのかを共有する

**担当: ALL(スクラムチーム全員)**

まず、私たちが「なぜこのプロジェクトをやるのか」「何を成し遂げたいのか」というビジョンをチーム全員ですり合わせます。インセプションデッキなどを作るのが効果的です。
この活動を通じて、チームが目指すべき長期的な目標である**プロダクトゴール**を明確にします。

### 2. やることをリスト化する (プロダクトバックログ作成)

**担当: ALL**

プロダクトゴールを達成するために必要なことをリスト（PBL: プロダクトバックログ）にしていきます。

0. **ペルソナ・ユーザージャーニーを作っておく**
   * ペルソナとは、想定ユーザーとそのバックグラウンドの情報のことです
   * ユーザージャーニーとは、ユーザーがそのプロダクトを使って目標を達成するまでの体験全体の流れのことです。
   * インセプションデッキをAIに渡せば作ってくれます
1. **アイデアを洗い出す**: やるべきこと、作りたいもの（PBI: プロダクトバックログアイテム）をユーザージャーニーを参考に付箋などに書き出します。
    * **補足: エピック**: この時点では、「ユーザー認証機能」のような大きな粒度（エピック）でOKです。エピックは、後でスプリントで扱える小さなPBI（ユーザーストーリーなど）に分割していきます。
2. **優先度をつける**: [緊急かつ重要・緊急ではないが重要・緊急だが重要ではない・緊急でも重要でもない・あれば嬉しい] などの基準でざっくり分類します。
3. **並び替える**: 分類を参考に、ビジネス価値の高い順、リスクの高い順などでPBIを並び替えます。特に最初の数スプリントで何から着手すべきか、重点的に考えます。
4. **POが承認**: 並び順にPOが最終的な責任を持ちます。
5. **完成の定義 (DoD) を決める**: このタイミングで「何をもって"完成"とするか」の基準（DoD: Definitions of Done）をチーム全員で決めます。（例: テストコードが書かれている、ドキュメントが更新されている、など）

### 3. 作業量を見積もる

**担当: Dev + (SM・PO)**

PBIの作業量を「時間」ではなく「**他の作業との比較 (相対的な大きさ)**」で見積もります。

0. **PBLの上位を詳細化する（リファインメント）**: 詳細は「スプリントの進め方」の章を参照。
1. **基準を決める**:
   * PBIをその作業量によって[大変・少し大変・簡単]に分類します。
   * 「少し大変」に分類されたPBIを作業量順に並べ、真ん中あたりの作業量が明確に分かるアイテムを基準とします。
   * このPBIを仮に作業量「3」とし、基準とします。
2. **プランニングポーカーで見積もる**:
    * フィボナッチ数（1, 2, 3, 5, 8, 13...）が書かれたカードを使います。
    * 開発者全員で、1つのPBIに対して基準と比べた作業量を考え、一斉にカードを出します。
    * 見積もりに大きな差があれば、その理由（考慮漏れ、技術的な懸念など）を話し合います。
    * 対話を通じて認識を合わせ、全員が納得するまでポーカーを繰り返します。

> **重要なのは:**
>
> * この見積もりは**約束ではなく推測**であること。不確実なことの予測に時間をかけすぎず、実際に作業した**実績**で未来の予測精度を上げていきます。
> * **全員参加と対話**を通じて、PBIに対する共通認識を深めることが一番の目的です。

### 4. いつ頃できそうか予測する (ベロシティ)

**担当: ALL**

ベロシティとは、**そのスクラムチームが1スプリントに完了できる作業量**のことです。PBIの見積もりで使用した作業量を使用し、例えば「1スプリントあたり20ポイント」などとして表します。

* **計測方法**:
  * 最初の1スプリントを終えると、最初の実績値がわかります。
  * **3〜4スプリントを経験し、その実績値の平均**を使うと、より信頼性の高い予測ができるようになります。
* **リリース計画**:
  * PBL全体のポイント合計を、チームのベロシティで割ることで、プロダクトゴール達成までのおおよその期間が予測できます。
  * POはこの予測を基に、期間を優先するならスコープ（PBLの範囲）を、スコープを優先するなら期間を、ステークホルダーと調整します。

> **Note:**
> スクラムに慣れていないチームではベロシティが安定しないことがあるため、POはそのことをあらかじめステークホルダーに伝えておきましょう。

## スプリントの進め方

**スプリント**とは、計画・開発・レビュー・改善のサイクルを回すための、**1〜4週間の固定された期間**です。私たちのチームでは、まず`[期間を入力]`で始めてみましょう。

### 0. PBLのリファインメント

**担当: PO**

* **目的**: 次以降のスプリントでスムーズに作業に入れるよう、PBIを常に準備された状態にしておくこと。
* **やること:**
  * **追加**: 新しい発見や要望を新しいPBIとして追加する。
  * **詳細化**: 優先度の高いPBIを、開発チームが**次のスプリントで着手できる**レベルまで具体的にし、タスクに分解しやすくする。(詳細は後述)
  * **優先順位の見直し**: ビジネス価値や学習効果に基づき、PBIの順番を入れ替える。
  * **削除**: もはや不要になった、価値がないと判断されたPBIを削除する。

> **Note:**
> PBRの詳細化における「着手できる」とは下記のような状態です。
>
> * 開発チームが**見積もりができるくらい**「何をすべきか」が具体的になっている。
> * **1つのPBIが1スプリント内で完了できる**サイズになっている。
> * 受け入れ条件が定義されている。

### 1. 計画: スプリントプランニング

**担当: ALL**

スプリントの最初に、これから何にどう取り組むか計画を立てます。スプリントプランニングは下記3ステップで実施します。

1. **Why (なぜやるのか)**: POが今回のスプリントで達成したいことを説明し、チームで**スプリントゴール**を決めます。
2. **What (何をやるのか)**: スプリントゴールを達成するために必要なPBIを、PBLから選択します。このPBIのリストが**スプリントバックログ**の骨子となります。
3. **How (どうやるのか)**: 開発者は、選択したPBIを作業に着手可能な粒度の具体的な**タスク**に分解し、作業計画を立てます。

### 2. 日々の作戦会議: デイリースクラム

**担当: Dev + (SM)**

毎日決まった時間・場所で、15分以内で開催する作戦会議です。

* **目的**: スプリントゴール達成に向けた進捗を確認し、問題があれば早期に発見して、その日の計画を調整すること。
* 単なる進捗報告会ではありません。
* 何か問題が見つかった場合は、デイリースクラムの場では深追いせず、必要なメンバーだけで別途集まって解決します。

### 3. 成果の確認: スプリントレビュー

**担当: ALL + ステークホルダー**

スプリントの終わりに、完成した成果物（動くソフトウェアなど）のデモを行います。

* DoDを満たしたものだけを「完成」としてデモします。「動くけどテストが終わってない」ものは未完成です。
* これは単なる進捗報告会ではありません。ステークホルダーから**フィードバックをもらい、対話を通じてプロダクトが正しい方向に進んでいるかを確認し、次の計画に活かす**ための場です。

### 4. ふりかえり: スプリントレトロスペクティブ

**担当: ALL**

スプリントの最後に、チーム自身を改善するために行う「ふりかえり」です。

* **目的**: プロセス、ツール、人間関係など、スプリントのあらゆる側面で「何が上手くいき、何が問題だったか」を話し合い、**チームをより良くするための具体的な改善策を見つける**こと。
* **進め方の例 (KPT)**:
  * **Keep**: 良かったこと、続けたいこと
  * **Problem**: 問題だったこと、やめたいこと
  * **Try**: 次に試してみたいこと
* ここで重要なのは、**次のスプリントで実践する改善アクションを1つか2つ、具体的に決める**ことです。

## Tips: ウォーターフォールに慣れた人へ

アジャイルは本質的に不確実な環境に立ち向かうためのフレームワークです。ウォーターフォールとは市場・事業環境に対する世界観が根本的に異なります。

ウォーターフォールの世界では「最初に立てた詳細な計画（WBSなど）が正しく、見積もりは**約束**」と考えます。

しかし、アジャイルの世界観は全く逆です。私たちは「未来は不確実で、計画は必ず変わり、見積もりはただの**推測**にすぎない」という現実からスタートします。

先の見えない不確実な状況下で私たちが唯一信頼できる武器が、「過去にこれだけ達成できた」という**実績値**、すなわちベロシティです。

この実績値を正しく未来の推測に使うために、見積もりは正直に行いましょう。
また、SMはベロシティをあげさせたり、スコープを不当に追加しようとする要求からチームを守る責務があります。POはこのことを認識し、SMと協力してステークホルダーとの対話を行いましょう。
