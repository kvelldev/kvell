# Inception Deck: Project Kvell

## 1. 我々はなぜここにいるのか (Why are we here?)

### 現代のインターネットには「B層（サイレントマジョリティ）」の居場所がない。
* **X (Twitter) は「個」が強すぎる。**
    * 「ファンである私」と「社会人の私」を切り分けたいのに、アカウントという「人格」が常に評価（いいね・フォロワー数）に晒される。
    * ファンダムの同調圧力やマウント合戦に疲れ、純粋にコンテンツを楽しめない。
* **5ch は「闇」が深すぎる。**
    * 匿名性は魅力的だが、アングラ感、誹謗中傷、過去のログが残り続ける恐怖（デジタルタトゥー）が、一般人の書き込みを躊躇させる。

### Mission
**「微熱の共有」と「後腐れのない熱狂」のためのサードプレイスを作る。**
人格を捨て、評価を捨て、ただ「その瞬間の感情」だけを薪としてくべられる場所を提供する。

## 2. エレベーターピッチ (Elevator Pitch)

* **「推し活」の同調圧力や承認欲求に疲れた、**
* **ごく普通の社会人ファン（B層）** 向けの、
* **Kvell** というサービスは、
* **「人格」を持たない焚き火チャット** である。
* これは **Xや5ch** とは違い、
* **「いいね」による個人評価を廃止し、投稿が時間とともに「灰」になって消滅する** ことが特徴である。

## 3. パッケージデザイン (Package Design)

* **プロダクト名:** Kvell（クベル / 他人のことについて喜ぶ・誇りに思う）※起案時の仮称: Takibi
* **キャッチコピー:** **「名前も、履歴も、評価もいらない。ただ、薪をくべろ。」**
* **デザインコンセプト:** "Night & Bonfire"
    * ダークモード基調。炎の揺らぎを感じさせるUI。
    * テキストは「ログ」ではなく「火の粉」として扱い、物理演算的にフローさせる。
* **キー体験:**
    * アプリを開くと、誰かがくべた火が燃えている（＝今、誰かが起きている気配）。
    * 書き込んでも「通知」は来ない。ただ、画面上の炎が少し大きくなるだけ。

## 4. やらないことリスト (NOT List)

**個人開発のリソースを守り、かつ「承認欲求」をシステム的に排除するための鉄の掟。**

* **× ユーザー登録・固定ID:**
    * ブラウザセッションベースの完全匿名。過去の発言と紐付かない。
* **× 「いいね」ボタン:**
    * **やる:** 「薪をくべる」ボタン。投稿者への賞賛ではなく、話題（スレ）への燃料投下とする。
* **× 過去ログの保存・検索:**
    * 「今」の熱狂のみに価値がある。終わった祭りの跡地は見せない。
* **× リプライ・引用RT:**
    * 議論や論破は不要。独り言が並列に流れるだけ。
* **× 画像アップロード:**
    * 管理コスト削減とリスク回避。ホワイトリストURL（X公式等）の展開のみ。

## 5. 技術的な解決策 (Technical Solution)

**コストを抑えつつ、リアルタイムの「熱狂」を演出する。**

* **Architecture:** Simple & Robust (AWS EC2 "One-Legged" approach)
    * **Frontend:** React (SPA/PWA) - S3 + CloudFront hosting.
    * **Backend:** FastAPI (Python) - EC2.
    * **DB:** MongoDB (Log storage is temporary/capped).
    * **Realtime:** WebSocket (FastAPI support).
* **Safety Valve (パケ死・負荷対策):**
    * CloudFront Geo Restriction（日本国内限定）。
    * 画像は外部URL展開のみで自鯖には置かない。
    * 同時接続数が閾値を超えた場合、WebSocketの更新頻度を間引く（炎のエフェクトはクライアントサイドで補間）。
* **Agile**
    * 本PJはアジャイルにて開発を行う。
* **AI-Native**
    * 個人開発のリソース不足を補うため、コーディングは基本的にAgentic AIに実施させる。
    * AIには下記をプロンプトとして渡し、PRの発行まで実施させる。
        * PBI(ユーザーストーリー)
        * アーキテクチャ要件のプロンプト(ガードレール)
        * PRテンプレート
    * AI生成物の品質担保は下記の方法で行う。
        * 静的解析で拾えるとこは全部拾う。
        * CIで自動テスト(BDD)・パフォーマンステスト(k6)・セキュリティスキャン(codeQL)を走らせる。
        * GherkinシナリオやPRのレビュー並びにマージは人間が行う。
        * Clean Architectureを採用し、AIによるスパゲッティコードの生成を抑制し、人間のレビュー負荷を下げる。

## 6. ターゲット・ペルソナ (The Persona)

**菊池健一郎（28歳・男性）**

* **職業/属性:** 大手メーカーの経理部。年収700万。彼女あり。
* **性格:** 協調性が高く、職場の飲み会も断らない「常識人」。ネットリテラシーは平均的だが、炎上リスクには敏感。
* **趣味:** **「櫻坂46」の隠れファン**。
  * 毎週日曜深夜の冠番組「そこ曲がったら、櫻坂?」の放送を楽しみにしている。仕事があるのでリアタイはせず、配信サービスで視聴する。
  * ライブには行くこともある。グッズはライブ用にペンライトのみ所有。
  * 職場やリアルの友人には「まあ、嫌いじゃないけど」程度に濁している（オタクと思われると社会的コストがかかるため。また、「オタクである自分」が理想の自分と一致しないため。）。
* **インサイト（隠れた欲求）:**
  * **「推しを支える」とか「運営に貢献する」とか、正直しんどい。** お金は払うから、ただ面白いものを見せてほしいというドライな消費者マインドを持っている。
  * 「推し活」アカウントを作って運用するほどマメではないし、スベるのが怖い。
  * でも、冠番組を見て心が動いた時、その**微熱**をどこかに吐き出したい。
  * 独り言をXに書いても、フォロワーである大学の友人に見られるのは恥ずかしいし、かといって誰も見ていないメモ帳に書くのでは虚しい。
  * パートナーはこの話題に興味がない。
  * 5chには情報がたくさんあるため見ることはあるが、スレの荒れ具合を見て嫌な気分になり、すぐ閉じてしまう。怖いので、書き込みは考えたこともない。
* **理想の状態:**
  * 誰かが始めた「祭り」に、匿名で薪を一本投げて、誰にもバレずにニヤニヤして、飽きたら無言で立ち去りたい。
  * 誰とも繋がらず、誰にも評価されず、ただ「今日の井上かわいかったな」という **"微熱"** だけを置いて眠りにつきたい。

## 7. コア・メカニクス：種火と焚き火 (Spark to Bonfire)

**ユーザーがスレ立てできない代わりに、システムが熱狂を可視化する。**

1.  **種火 (Spark):**
    * 全ての投稿はまず「種火」としてTLを流れる。
    * 公式ニュースもBotが「種火」として投下する。
2.  **薪 (Fuel):**
    * ユーザーは共感した種火に「薪」ボタンを押す。
    * これは投稿者への評価ではなく、**「この話題をもっと燃やせ」** というシステムへの投票。
3.  **焚き火 (Bonfire):**
    * 短時間に大量の薪がくべられた種火は、**自動的に「焚き火（スレッド）」に昇格**し、画面上部に固定される。
    * 焚き火内での投稿は、さらなる燃料となり、炎のエフェクトを最大化させる。
4.  **鎮火 (Fade out):**
    * 薪が供給されなくなると、焚き火は徐々に小さくなり、最終的にログごと消滅する。

## 8. 夜も眠れない問題 (Nightmares)

* **過疎:**
    * 特に、大規模な広告を打てない個人開発では初期のユーザー集めが鬼門となるはず。
    * 「スレ立てできない」は対立煽りや承認欲求ありきの「サムい」話題を乱立させたくないことへの回答だが、過疎問題に対して諸刃の剣となる可能性がある。
    * 対策: Botによる公式ニュース投下で、最低限の「種火」は常に供給する。
* **無法地帯化:**
    * 対策: 「見たくない（スワイプ）」数による自動シャドウバン。裏側でのIP記録（法的対応用）。
* **コスト超過:**
    * 対策: AWS Budgetsによる監視と、EC2インスタンスサイズの固定。
