# プロジェクト企画書：Kvell

v0.3

## 1. プロダクト・ビジョン

**「微熱の共有」と「後腐れのない熱狂」**
既存SNS・BBSが強制する「常時接続・相互評価・監視」からユーザーを解放し、**「同じ火（話題）を見つめるアノニマスな隣人」** としての心地よい「場」を提供する。

## 2. コアバリュー（Anti-Thesis & Value Proposition）

本サービスは、既存SNS/BBS(特にXや5ch)を使う「ファンであること=本当の自分」というアイデンティティを持つユーザー（A層）に対し、**「『ファンであること』はその人にとってあくまでつけ外し可能なオプションであるサイレントマジョリティ（B層）」** のための場所として価値を定義する。

### ■ Anti-X （脱・承認欲求と評価経済）

* **【課題】** X（旧Twitter）は「個」が強すぎる。発言は「誰が言ったか」で判断され、いいね数は「個人のスコア」として機能する。本来無責任な楽しみであるはずの「推し活」ですら、ファン同士のマウント合戦や、ファンダムによる「正しく推せているか」という同調圧力に晒される。
* **【解決策】** **「『私』と『ファンである私』の完全な分離」と「個の評価の廃止」**。
  * セッションごとのIDリセットにより、「名無し」による投稿のみを許可する。過去の発言や人格を積み上げさせない。
  * 投稿への「いいね（=個人の手柄）」を廃止し、**「薪（話題への燃料）」** のみを採用。
  * **価値:** 「良きファン」である努力を放棄できる。**「通りすがりの無責任な野次馬」** として、リスクも責任も負わず、純粋にコンテンツを楽しむ権利を取り戻す。承認欲求から来る、質の低い・過度に煽動的な投稿をみる可能性を下げることができる。

### ■ Anti-5ch （脱・コンテキストと恐怖）

* **【課題】** 5chはアングラ感が強く、一般人には「書き込んだら攻撃される」「開示請求される」という恐怖(偏見)がある。また、「スレ立て」等の自治ルールが厳しく、ライト層が気軽に参加できない。また、2000年代初頭からの世間的なイメージから、公共の場で開くことが恥ずかしい(「この人5chねらーなんだ」という視線)。
* **【解決策】** **「フロー特化（忘却）」と「クリーンなUI」**。
  * 投稿ログは物理的に消える（証拠が残らない）。リプライ機能を廃止し、論争をシステム的に不可能にする。
  * **価値:** 「叩かれるリスク」と「論破されるリスク」をゼロにし、知識のないライト層でも気軽に「〇〇かわいい」と書き込める参加の容易さ。

## 3. ターゲット・ペルソナ詳細

**菊池健一郎（28歳・男性）**

* **職業/属性:** 大手メーカーの経理部。年収700万。彼女あり。
* **性格:** 協調性が高く、職場の飲み会も断らない「常識人」。ネットリテラシーは平均的だが、炎上リスクには敏感。
* **趣味:** **「櫻坂46」の隠れファン**。
  * 毎週日曜深夜の冠番組「そこ曲がったら、櫻坂?」の放送を楽しみにしている。仕事があるのでリアタイはせず、配信サービスで視聴する。
  * ライブには行くこともある。グッズはライブ用にペンライトのみ所有。
  * 職場やリアルの友人には「まあ、嫌いじゃないけど」程度に濁している（オタクと思われると社会的コストがかかるため。また、「オタクである自分」が理想の自分と一致しないため。）。
* **インサイト（隠れた欲求）:**
  * **「推しを支える」とか「運営に貢献する」とか、正直しんどい。** お金は払うから、ただ面白いものを見せてほしいというドライな消費者マインドを持っている。
  * 「推し活」アカウントを作って運用するほどマメではないし、スベるのが怖い。
  * でも、冠番組を見て心が動いた時、その**微熱**をどこかに吐き出したい。
  * 独り言をXに書いても、フォロワーである大学の友人に見られるのは恥ずかしいし、かといって誰も見ていないメモ帳に書くのでは虚しい。
  * パートナーはこの話題に興味がない。
  * 5chには情報がたくさんあるため見ることはあるが、スレの荒れ具合を見て嫌な気分になり、すぐ閉じてしまう。怖いので、書き込みは考えたこともない。
* **理想の状態:**
  * 誰かが始めた「祭り」に、匿名で薪を一本投げて、誰にもバレずにニヤニヤして、飽きたら無言で立ち去りたい。
  * 誰とも繋がらず、誰にも評価されず、ただ「今日の井上かわいかったな」という **"微熱"** だけを置いて眠りにつきたい。

## 4. システム概念図：種火システム (Spark to Bonfire)

**【設計思想：いいね vs 薪】**
「いいね」は投稿者個人への報酬であり、承認欲求を刺激するが、同時に「評価されるプレッシャー」を生む。
本システムにおける「薪」は、**投稿者ではなく「その話題（スレ）」に対する燃料**である。ユーザーは投稿者を称賛するのではなく、「その話題もっと燃えろ」という応援の意味でボタンを押す。

1. **種火 (Spark):** ユーザーの投稿（独り言）、およびAPI取得されたニュース。
2. **着火 (Ignition):** 短時間に一定数のリアクション（薪）が集まった種火のみが「スレッド（焚き火）」に昇格する。
3. **鎮火 (Decay):** 反応のない種火、および時間が経過したスレッドは、文字が薄くなり消滅する。

## 5. 画面概念図(UI/UX Concept)

### 5.1. 画面レイアウト定義 (Mobile View)

理想形

```text
+--------------------------------------------------+
|  [Status]  　炎が燃え上がっています                |  <- A. プレゼンス・バー
+--------------------------------------------------+
|                                                  |
|   (背景：揺らぐ炎のエフェクト)                   |
|                                                  |
|        【 櫻坂  】     【  新曲  】              |  <- B. 焚き火エリア (Main Stage)
|                                                  |
|    「今日の衣装マジで天使すぎた...」             |
|                                                  |
|           (ボワッと光るテキスト)                 |
|                                                  |
+--------------------------------------------------+
|   ---------------- (境界線) ------------------   |
+--------------------------------------------------+
|  ID_x: きたああああああああ                    |  <- C. 種火エリア (Sub Stream)
|  ID_y: https://x.com/stat... [画像展開]        |
|  ID_z: 演出神かよ                              |
|  ID_w: 泣いた                                  |
|  ID_v: まだ始まって10分だぞｗ                  |
|                                                  |
|   (下から上へ、高速でフローしていく)             |
|         [更に種火を見る]                   |
+--------------------------------------------------+
| [  種火を投げる...                  ] [ 投げる ] |  <- D. 投稿バー
+--------------------------------------------------+
```

### 5.2. 各コンポーネント詳細

#### A. プレゼンス・バー (The Presence)

* **表示:** 具体的な「いいね数」などの評価指標は置かず、**「比喩的な同接人数」** のみを表示する。
* **比喩表現例**
  * 〜10人: 「小さな火が灯っています」
    * （過疎ではなく「あえて静か」なニュアンス。チルな雰囲気）
  * 〜100人: 「暖かな空気が流れています」
    * （これがデフォルト。居心地の良さを強調）
  * 〜1000人: 「炎が燃え上がっています」
    * （勢いが出てきた状態。熱量を感じさせる）
  * 1000人〜: 「夜空を焦がす勢いです」
    * （圧倒的な熱狂。非日常感）

#### B. 焚き火エリア (The Bonfire / Upper Half)

* **役割:** 「今、コミュニティで最も熱い話題」を表示するメインステージ。
* **挙動:**
  * **生存競争:** 複数の種火から「薪」を多く集めた投稿、またはシステムが抽出した「ホットワード」が大きく表示される。タップするとスレッドビューに移行する。
  * **鎮火:** 熱量が下がるとテキストは徐々に透明になり、やがて消滅。

#### C. 種火エリア (The Spark Flow / Lower Half)

* **役割:** 全ユーザーの投稿がリアルタイムに流れる川。
* **挙動:**
  * **フロー:** 新しい投稿が下から現れ、上（焚き火エリアとの境界）に向かって消えていく。
  * **画像展開:** ホワイトリスト（X, Instagram等）に含まれるURLは、このエリア内でサムネイル展開される。信頼できないドメインを展開しないことで、グロ画像等による荒らし対策とする。
  * **薪をくべる:** ユーザーは流れていく投稿に「薪」をくべることができる。タップ時、火の粉が舞うエフェクトが発生する（投稿者への通知は行わない）。

#### D. 投稿バー (Action)

* **役割:** 種火を放流するための入力エリア。
* **制限:**
  * 画像アップロードボタンなし。
  * 投稿はテキストのみで、URLはパースする。
  * リプライ/引用ボタンなし。
* **体験:** 送信ボタンを押すと、即座に自分の投稿が「C. 種火エリア」の最下部に挿入され、全ユーザーの画面を流れていく。

### 【EPIC 6: リスク管理・運営守備】 (The Defense)

**前提:** ユーザーの安心を損なわないよう、これらの機能は**極力目立たない場所（フッター、設定メニューの奥など）**に配置しつつ、法的な効力は確保する。

|:---|:---|:---|:---|:---|
| **US-13** | **システム管理者** | 投稿がUIから消えた後も、**「IPアドレス」「投稿日時」「投稿内容」「UserAgent」をデータベースの非公開領域に一定期間（30日程度）保持したい。 | 殺害予告や権利侵害などの法的トラブルが発生した際、捜査機関や開示請求に適切に対応し、管理者自身の法的責任（共犯扱い等）を回避するため。 | 1. フロントエンド（ユーザー画面）へのAPIレスポンスには、絶対にIPなどの個人情報を含めない。<br>2.
