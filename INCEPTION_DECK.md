
# Inception Deck: Project Kvell

v1.0

## 1. 我々はなぜここにいるのか (Why are we here?)

### 「推し活」という言葉の、押し付けがましさが嫌いだ

世の中は「推し活」ブームだ。対象を「推す（支える）」ことが正義とされ、ファンであることはアイデンティティの一部となり、SNSでは「いかに愛しているか」の証明合戦が行われている。

**でも、僕は「推し」たいわけじゃない。ただ、「好き」でいたいだけだ。**

僕にとって、アイドルファンであることは「自分そのもの」ではない。社会生活を送る僕にとっての、**つけ外し可能なオプション**に過ぎない。
リアルな友人や同僚とこの話題を話さないのは、隠したいからではない。仕事や恋愛の話のほうが盛り上がるからだ。大人はTPOに合わせて、最適な話題と場所を選び取る。

### Current Problem

この「ファンというオプション」を快適に起動できる場所が、現代のネット上にはない。

* **X (Twitter):** 「推す」ことがシステム的に強制されている(ドーパミンのハック)。ノイズが多く、承認欲求のマウント合戦に疲れる。
* **5ch:** 文脈は合っているが、毒が強すぎる。書き込みへの恐怖（開示請求、古参の叩き）が、一般人の参加を阻む。

### Our Mission

**ユーザー同士が相互に作用する創発的な空間を、ノイズのない空間でシステマチックに作り出す。**

僕が欲しいのは、誰の目も気にせず、無責任に「今日の〇〇かわいかったな」という**微熱**を吐き出せる場所。
そして、その微熱が他人の共感を呼び、自然発生的に大きな **「焚き火（祭り）」** となって燃え上がり、終われば綺麗な **「星座（伝説）」** として空に残る場所だ。

僕たちは、 **「ファンという着ぐるみ」を安全に着脱でき、ゴミ（駄文）と宝石（名スレ）を自動選別する「熱狂の精製装置」** を作る。

## 2. エレベーターピッチ (Elevator Pitch)

* **「推し活」の同調圧力や承認欲求に疲れた、**
* **ごく普通の社会人ファン** 向けの、
* **Kvell** というサービスは、
* **「ゴミと宝石」を自動選別する匿名掲示板** である。
* これは **Xや5ch** とは違い、
* **「いいね」や「返信」による評価・議論を廃止し、共感された投稿だけが「焚き火」として燃え上がり、最終的に「星座」としてアーカイブされる** ことが特徴である。

## 3. パッケージデザイン (Package Design)

* **プロダクト名:** Kvell（クベル / 他人に対する喜び・誇り）
* **キャッチコピー:** **A BBS for silent majority.**
* **デザインコンセプト:** "Night, Bonfire & Constellation"
  * **Input:** 独り言をつぶやく「夜の闇（没入型モーダル）」。
  * **Active:** 炎が重なり合う「焚き火（パイル表示）」。
  * **Archive:** 燃え尽きた後に残る「星座（殿堂入りリスト）」。

## 4. やらないことリスト (NOT List)

**「荒れ」と「承認欲求」を構造的に排除し、ROM専ユーザーを守るための鉄の掟。**

* **× ユーザー登録・固定ID・コテハン:**
  * ブラウザセッションベースの完全匿名。過去の発言と紐付かない。
* **× 「いいね」数・フォロワー数の表示:**
  * 数字による評価は承認欲求を生み、ノイズとなるため一切表示しない。
* **× リプライ・ツリー表示:**
  * 特定の個人への「返信」機能は持たない（レスバの防止）。あるのは話題への「追焚き（Add Fuel）」のみ。
* **× ユーザーによる「スレッド作成」:**
  * ユーザーが恣意的に「場」を作ることはできない。種火を投げることしかできない。
* **× 通知 (Notification):**
  * 自分の投稿がどうなったか、通知は来ない。荒らしへの報酬（反応）を与えない。
* **× 画像アップロード:**
  * 管理コスト削減とリスク回避。ホワイトリストURL（X公式等）の展開のみ。

## 6. コア・メカニクス：Spark, Bonfire, Constellation

**自然淘汰による「浄化」と「資産化」のサイクル。**

1. **種火 (Spark) - [Flow]**
    * ユーザーは「種火（短文の独り言）」のみ投稿可能。
    * **Input UX:** チャットではなく「独白」。誰かに話しかけるのではなく、夜空に吐き出す体験。
    * **Lifecycle:** 寿命は極めて短い（例：10分）。誰からも「追焚き」されなければ、人知れず消滅する（ゴミの自動廃棄）。
2. **薪をくべる/追焚き (Add Fuel) - [Action]**
    * **薪をくべる**: 共感した種火に対し、薪をくべる(共感ボタンを押す)。薪数は表示されない。
    * **追焚き**: 共感した種火に対し、薪をくべる（引用して独り言を重ねる）。
    * **UI:** 返信ツリーではなく、**薪の山（Pile）** として視覚化。
3. **焚き火 (Bonfire) - [Active]**
    * **Black Box Algorithm:** 「ユニークユーザー数」「追焚き速度」などが一定閾値を超えた種火のみが **「着火」** 判定される。このアルゴリズムは公開しない。
    * 着火するとトップページに固定され、全ユーザーが参加する「祭り会場」となる。
4. **星座 (Constellation) - [Stock]**
    * 祭りが終わった焚き火の中で、特に質が高かったものは**「星座（殿堂入り）」**としてアーカイブされる。
    * ROM専ユーザーのための読み物コンテンツとなり、SEO流入の受け皿となる。
    * 事後のアクションは「見上げる（軽い足跡）」のみ可能。

## 7. ユーザーフロー

基本は「星座閲覧->焚き火閲覧&レス->種火投稿」のような学習曲線を描く想定。
1:9:90の法則から言って、種火投稿のハードルはかなり高いはず。
種火投稿なしでもリテンションを維持できるように、UX的な作り込みがある程度必要。

1. **Entry (星座体験):**
    * ユーザーはまず、**運営が用意した「伝説の過去ログ（北極星）」** を読む。
    * UIは「アーカイブ一覧」ではなく「夜空の星」として美しく見せる。
2. **Hook (物理的フィードバック):**
    * 現在進行系の焚き火や種火に「薪」を投げると、**炎が視覚的に反応する（ボッと燃える）**。
    * これにより「参加した」という微かな快感（貢献感）を得る。
3. **Action (種火の成仏):**
    * 勇気を出して種火を投げ、もし誰も反応しなくても、**「夜空に静かに溶けていく」美しい消滅アニメーション** を見せる。
    * 「無視された」ではなく「夜に預けた」という体験にする。

## 8. 技術的な解決策 (Technical Solution)

とにかくコストを抑える。

* **Architecture:** Simple & Robust (AWS EC2 "One-Legged" approach)
  * **Frontend:** React (SPA/PWA) - S3 + CloudFront hosting.
  * **Backend:** FastAPI (Python) - EC2.
  * **DB:** MongoDB (Log storage is temporary/capped).
  * **KVS:** Redis
  * **Realtime:** WebSocket (FastAPI support).
* **Safety Valve (パケ死・負荷対策):**
  * CloudFront Geo Restriction（日本国内限定）。
  * 画像は外部URL展開のみで自鯖には置かない。
  * 同時接続数が閾値を超えた場合、WebSocketの更新頻度を間引く（炎のエフェクトはクライアントサイドで補間）。
* **Agile**
  * 本PJはアジャイルにて開発を行う。
* **AI-Native**
  * 個人開発のリソース不足を補うため、コーディングは基本的にAgentic AIに実施させる。
  * AIには下記をプロンプトとして渡し、PRの発行まで実施させる。
    * PBI(ユーザーストーリー)
    * アーキテクチャ要件のプロンプト(ガードレール)
    * PRテンプレート
  * AI生成物の品質担保は下記の方法で行う。
    * 静的解析で拾えるとこは全部拾う。
    * CIで自動テスト(BDD)・パフォーマンステスト(k6)・セキュリティスキャン(codeQL)を走らせる。
    * GherkinシナリオやPRのレビュー並びにマージは人間が行う。
    * Clean Architectureを採用し、AIによるスパゲッティコードの生成を抑制し、人間のレビュー負荷を下げる。

## 9. 夜も眠れない問題 (Nightmares) & 対策

* **過疎（The Empty Sky）:**
  * ユーザーが少ないと、空に何も浮かんでいない（種火がない）状態になる。
  * **対策:** **「北極星 (Polaris)」システム**。
    * 運営Botが「公式ニュース（テレビ出演情報など）」を定期的に種火として投下する。
    * この種火だけは寿命が長く、常に最低限の「明かり」と「語る場所」を提供する。
* **荒らしによるハック:**
  * 組織票で無理やり「焚き火」を作ろうとする。
  * **対策:** 焚き火判定アルゴリズムのブラックボックス化と、同一IP/Fingerprintからの連続投稿に対するウェイト下げ。
* **法的リスク:**
  * **対策:** 匿名だが、サーバーサイドではIPとタイムスタンプを法的保存期間保持する。通報機能による非表示対応。
